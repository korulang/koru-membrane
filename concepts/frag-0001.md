A koru `[comptime|transform]` can lower a call site to a **real event
invocation** — not only to an inline Zig body. This is a supported, working
path through the transform pass runner, and it is the koru-native way for a
compile-time DSL to lower to *composable flow* rather than opaque host code.

The prior working belief (held implicitly by every transform in the corpus at
the time) was narrower: a transform generates a string of Zig and stuffs it
into `Invocation.inline_body`, and the emitter drops that verbatim at the call
site. `fmt.blk`, `ln`, and `sqlite3:query` all do exactly this — they build an
`inline_code` string and reroute to an `.impl` path carrying it. Nothing had
been shown generating a *real* invocation of a different event.

The `view` DSL tracer proved the wider path. Its proc returns
`.{ .replacement = Item{ .flow = ... } }` whose `body` is an `ast.rootSite`
over an `ast.Invocation` that targets a **different runtime event**
(`koru/vaxis:write-at`), with real `ast.Arg`s and **no `inline_body`**. The
transform pass runner re-emits that generated invocation through write-at's own
normal shape-check + codegen — the emitted Zig is
`write_at_event.handler(.{ .x = 0, .y = 0, .text = "hi" })`, indistinguishable
from a hand-written `write-at` call. `koruc build` compiled it; "hi" painted on
screen. (Evidence lives in koru-libs `vaxis/index.kz` `view` proc + the
`view: markup lowers to a real write-at invocation` commit — referenced, not
restated, because code moves and prose lies.)

Why this matters beyond a single element: markup like
`view { <box><text>{{name}}</text></box> }` can lower to a *chain of real koru
invocations* (`box`, `write-in`, `write-at`) that compose with everything else
koru knows — store `watch`, obligations, purity — instead of being sealed
inside a hand-emitted Zig block. The DSL becomes a source-to-flow rewrite, and
koru's binding/scoping rules never have to change.

**The chain is proven too (2026-07-22, same session).** N sibling `<text>`
elements lower to N real `write-at` invocations wired as a linear `|>` pipeline
— built as a nested `branch=""`/`kind=terminal` continuation chain, the exact
shape the parser produces for `a |> b |> c`, with the site's own trailing
continuations riding the last link. `koruc build` emitted three sequential
`write_at_event.handler(...)` calls at `y:0/1/2`; Lars confirmed all three lines
paint. So the earlier worry — that a *linear pipeline of N generated
invocations* (vs. `rootSite`'s single node) might surface the first real gap —
is answered: no gap. Multi-invocation flow-building from a transform works.

Open questions (not yet proven, each an increment on the chain):
- Whether a transform-generated invocation can itself target *another*
  transform (nested lowering, e.g. a generated `<box>` containing generated
  `<text>`), and whether re-triggering is correctly guarded.
- Whether the generated invocations can carry binds (`box(...): a |> write-in(win: a ...)`)
  across the generated chain — the retained-layout payoff needs it.
- Whether `{{ name:s }}` interpolation inside an element can reuse fmt's
  placeholder lowering so the text pulls from call-site scope.
