The vaxis "fmt renders the wrong text" bug was never about **allocation** — it
was about **byte ownership at render time**, and the fix belongs in the library,
not the language.

libvaxis is a RETAINED renderer that BORROWS. `Screen.writeCell` stores the cell
by value (`self.buf[i] = cell`) and `Cell.grapheme` is a `[]const u8` slice — so
the screen buffer holds a POINTER into the caller's bytes, and reads them only
later, at the deferred `render()`. Any text handed to `write-at` must therefore
stay alive and unchanged from the write until the frame renders.

Two callers violate that, and both look like "fmt is broken" but aren't:
- `std/fmt:ln` writes into a REUSED threadlocal buffer. A for-each firing one
  `ln` + `write-at` per row has every row's cell borrowing the SAME buffer; by
  render time it holds only the last row's text → every row shows the final
  value. (This is the `list_vaxis` "all-FINAL-value" bug.)
- `std/fmt:blk` returns `string<allocated!>` whose auto-discharge FREES the bytes
  at the end of the producing flow — before render → use-after-free.

The earlier framing (baton) treated this as a lifetime bug needing store-backed
data or a language `retain` feature. Both would work, but they over-reach: the
real, minimal fix is that **a retained renderer must OWN the bytes it will read
later.** `write-at`/`write-in`/`write-styled` now `dupe` their `text` into a
vaxis-owned arena (`text_arena`, reset on `clear()`) and hand `writeCell` slices
of that copy. The caller's buffer/obligation can be reused or freed the instant
the write returns. `string<allocated!>` is then a NON-issue — the copy happens
before the discharge — so fmt (any flavor) composes into views freely.

The general rule: **when a downstream stores a borrow and reads it later, the
borrow's lifetime is the STORER's problem to solve by owning a copy — not the
producer's problem to solve by living longer.** koru's obligation model was
correct the whole time; vaxis just had to own its pixels. (Evidence: koru-libs
`vaxis/index.kz` write-* procs + `text_arena`; the `list_vaxis` example renders
three distinct rows — referenced, not restated.)

Open: the arena resets on `clear()`, which is correct for full-redraw frames
(the reactive pattern) and merely grows-bounded for write-once static screens; a
long-running app that rewrites cells WITHOUT clearing would grow the arena
unbounded — revisit if that pattern appears. Relates to [[frag-0002]] (the same
"a retained/synthesized thing must self-provide what the normal path provides").
